As this is a recursive method, we start with our full set of features.

    plt.ylabel("Cross validation score (# correct classifications)")

Feature Analysis¶

It's easy to see that the recursive feature elimination curve is incredibly variable and not very intuitive. There are many potential reasons for this, but the two most likely would seem to be covariation and outlier distortion. It may be necessary to use PCA to compress related variables into a single vector or, if that fails, to vectorize term frequencies within the email corpus itself.

Our selected 'best features' have relatively poor p-values and low accuracy scores. Rather than depend on our recursive feature selection now, I chose to hand-pick three variables that, based on knowledge of the Enron case, would seem to correlate well with POI identification. Using these features, I can analyze any issues with our data.

Classifier Testing¶

Now that we have strong features to use in identifying our POIs, we will create and fit some classifiers. I fit five different classifiers as well as a 'soft' voting classifier, which takes the probabilities of the previous five fits and uses them to make a combined prediction.

As StratifiedShuffleSplit() is used in the grading program, I have used that same method for cross-validation. Stratified CVs are especially useful in the case where targets are relatively rare, as our POIs are. A tally is created for the best classifier (identified by F1 score) of true/false positives/negatives in order to produce precision and recall scores.

Classifier Tuning¶

These results were something of a lesson for me - I had originally spent a lot of time setting up a complex searching algorithm using both VotingClassifier and GridSearchCV in order to optimize my F1 score. To my chagrine, I found that the simplest classifier contained within the VotingClassifier was actually getting the best results, and that the other tune-able classifiers were actually dragging the F1 score down, particularly the SVC. The GridSearchCV also exponentially increased the time it took for the program to run.

For the sake of illustrating my thought process, I'll include a sample set of parameters I had used for a GridSearchCV.

I dared not include more than a pair per classifier attribute - the cross-validator already ran each classifier 1,000 times, and each set of values would require another 1,000-fold cross-validation. This set of parameters took about 5 hours to run. The GaussianNB() on its own takes somewhere close to ten seconds. Despite the tremendous increase in processor time, my F1 score in my VotingClassifier never came close to the default GaussianNB() score.

Finally, I ran the Udacity testing script to verify the classifier met scoring standards.

Concluding Q&A¶

Summarize the goal of this project - give some background on the dataset and how it can be used to answer the project question. Were there any outliers in the data when you got it, and how did you handle those?¶

This project involved using a variety of financial and email information to identify relatively rare individuals within our dataset (people of interest). I used supervised machine learning algorithms to train classifiers of a few different varieties. These algorithms were trained on features selected with a recursive feature elimination algorithm as well as hand-picking features based on qualititive knowledge of the Enron case.

There was one consistent outlier across many feature categories - the 'TOTAL' dictionary entry. It was easily identified once the data were visualized in boxplots. I eliminated the 'TOTAL' entry from the dataset, then checked each dictionary entry to ensure that no other totaling entry existed.

There were still outliers in the technical sense after this process - however, they were often the targets of our algorithm and therefore desireable to keep.

What algorithm did you end up using? What others did you try? How did model performance differ between algorithms?¶

The sklearn Gaussian Naive Bayes classifier ended up being my strongest classifier for this problem. I also fit a gradient boosting algorithm, an AdaBoost algorithm (decision tree base estimator), a random forest classifier, an SVC with 'rbf' kernel, and a 'soft' voting classifier that incorporated probability information from each to categorize points.

To my surprise, GaussianNB() outperformed each other classifier significantly. The F1 scores for each classifier are reported in the body of this document, with GaussianNB() having nearly double the F1 score of each other classifier.

What does it mean to tune the parameters of an algorithm, and what can happen if you don’t do this well? How did you tune the parameters of your particular algorithm?¶

Algorithm tuning parameters allow the programmer to optimize the algorithm for his or her needs. For example, support vector machines use a 'penalty' parameter, 'C', that affects how closely fit the classifier will be to the data.

Parameters fundamentally change the shape of the decision space of the classifier, but ultimately, they are subject to bias-variance tradeoff. If your classifier fits your training data too well and cannot generalize, you have 'overfit' the classifier. This is where the finesse of parameter tuning happens.

I had originally implemented a GridSearchCV parameter search to try to optimize my parameters, but even after quite a bit of work, the default GaussianNB() remained the strongest algorithm.

What is validation, and what’s a classic mistake you can make if you do it wrong? How did you validate your analysis?¶

Cross-validation is a primary means of protecting yourself from overfitting your classifier or biasing it incorrectly based on an unlikely random assortment of the points. Cross-validation (in this case, StratifiedShuffleSplit) runs the classifier a number of times on the same dataset, but uses different configurations of the data to protect against using a classifier that doesn't actually work well with the data.

A 'stratified' cross-validation method is used in this study because there are relatively few POI observations - the stratification keeps the proportion of non-POI and POI observations consistent between validation interations. If stratified validation is not used, many of the validations will be performed without POI observations at all.

Give at least 2 evaluation metrics and your average performance for each of them. Explain an interpretation of your metrics that says something human-understandable about your algorithm’s performance¶

Our classifiers must be rated based on how often they correctly classify data. As our classification problem for this project is binary (POI or not), there are only four possible classification outcomes, two good and two bad:

  1. True Positive (Identified as POI, Is POI)
  2. True Negative (Identified as non-POI, Is non-POI)
  3. False Positive (Identified as POI, Is Not POI)
  4. False Negative (Identified as non-POI, Is Not non-POI)

True Positives and Negatives are what we seek to maximize - correct identifications of POIs and non-POIs. Combinations of these values allow us some insight into the success of our classifiers. Precision, for example, is the proportion of true positives to the sum of true and false positives - precision is a measure of how well our algorithm can tell the difference between POI and non-POI. Recall, on the other hand, is a measure of how sensitive our algorithm is to finding POIs at all, which is to say, out of all POI's, what proportion were found as true positives. The composition of these two metrics gives the classifier's F1 score. Our best classifier's scores are as follows:
